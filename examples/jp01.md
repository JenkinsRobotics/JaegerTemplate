# Example — JP01 (the real one, as reference)

**Verdict: exists ✓ substantially today (voice, motion, lights, vision
v2, e-stop chokepoint are real and running) — the JP01 pass adds
contract adoption, out-of-tree loading, the capability-layer gap
closures, and the unit handshake, all already designed, none yet
shipped.**

Unlike the other five, this isn't a hypothetical mapping — it's the
one real hardware Jaeger in the ecosystem today, included so the other
five examples have a ground truth to be honest against. Every "exists"
claim below traces to a real file; every "planned" claim traces to a
named roadmap item, not a guess.

## What exists today

- **Voice.** `kokoro_tts` (`slot: tts`) + `whisper_stt` (`slot: stt`),
  both real shipped modules, both usable with or without JP01 attached
  — voice doesn't depend on the body.
- **Motion.** `mc01` controller, `topology.yaml`-declared:
  `motion.move_joints`, `motion.drive`, `motion.stop`
  (`allow_when_latched: true`), `motion.status` — all `HARDWARE` tier
  except status. Every field (ports, bauds, the Jetson-relay path) is
  survey truth from a real live measurement (2026-06-12), every
  controller still `simulated: true` until the operator walks it live.
- **Lights.** `avc01` controller: `lights.set_mode`, `lights.set_frame`,
  `lights.brightness` (`WRITE_LOCAL` — usable during an e-stop latch,
  deliberately, since status signaling should survive a stop),
  `lights.status` (`READ_ONLY`).
- **Vision v2.** `vcc01` controller: `robot_vision.stream_info`
  (`READ_ONLY`) — stream endpoint directory only; snapshot/detections
  are explicitly **(planned)**, landing when the live VCC01 path is
  walked (labelled honestly in the topology file's own comments, not
  glossed over).
- **The e-stop chokepoint.** `EStopLatch` sits at both the capability
  dispatcher and the transport chokepoint, independent of whether the
  agent loop is even running — the nervous-system rule enforced, not
  promised. `estop_scope: [mc01]`; `firmware_watchdog_required: true`
  (the one known real gap: MC01 has no L0 hardware watchdog yet, boot
  warns until firmware ships).
- **The capability layer's core mechanism.** Contrary to what you'd
  guess from "JP01 needs a pass" — `register_package_capabilities()`,
  the per-turn tool-catalog refresh, and fail-closed per-controller
  availability are **already built and already more complete than the
  engine-module→tool path**. "Plug in JP01 → agent gains motion tools"
  is a wiring problem today, not a framework problem — see the three
  gaps below, all scoped, all designed, none requiring new mechanism.

## What the JP01 pass adds (planned, all traced to a named design doc)

Per `dev/docs/roadmap/JROS_0.8_CAPABILITY_LAYER_DESIGN.md`:

1. **Gap 1 — no boot root ever loads the package.**
   `jaeger.toml`'s `hardware_jp01` node is `enabled = false` today;
   nothing calls `boot_hardware("jp01")` outside tests. Fix: flip it
   to a normal supervised manifest node, per-station (not global) —
   the Mac mini actually driving JP01 enables it, a dev laptop
   doesn't. Longer-term: the package gains its own `module.yaml`
   (slot `hardware`, multi-module slot) so discovery is uniform with
   engine modules — `topology.yaml`'s `capabilities:` block stays the
   richer tool-definition source either way.
2. **Gap 2 — the beta gate is the wrong gate.** Every hardware
   umbrella is `beta=True` today (invisible unless `JAEGER_DEV_MODE`)
   — a global dev switch standing in for a per-robot safety question.
   Fix: a per-**unit** `verified` flag (`unit.yaml`, flipped by
   `jaeger hardware verify jp01` after a guided live walk), not a
   global flag — unverified stays beta-gated exactly as today, verified
   becomes a normal fail-closed/e-stop-latched/permission-tiered tool
   (tier gating is the real runtime protection either way; this only
   changes *visibility*).
3. **Gap 3 — the Body has no identity record.** The Mind has
   `identity.yaml`; JP01 has nothing equivalent. Fix: `unit.yaml`
   (serial, `unit_id`, model, per-controller firmware versions,
   `verified`) plus a boot-time handshake — read unit.yaml, cross-check
   against `topology.yaml`'s `model`, query the live unit, refuse to
   register capabilities on mismatch. This is also the seam that later
   lets two bodies share one bus without collision (`unit_id`-prefixed
   health topics).
4. **Contract adoption + out-of-tree loading** (`THREE_TIER_STRUCTURE.md`
   0.9 work items 1 and 3): JP01_Firmware imports/vendors
   `jaeger_os/contract/` instead of keeping a second copy of topic
   names/ports/formats (retiring the exact "two copies of one truth"
   bug class that caused real field-week bugs); the hardware package
   loads from the JP01_Firmware repo as the first externally-loaded
   module, once out-of-tree module loading lands.

## Tiers used

PACKAGE/WORKSPACE (the hardware package IS the workspace, per
`TAXONOMY.md`) + MODULE (voice, independent of the body) +
DISTRIBUTION (JaegerAI running headless-configured, per
`JAEGER_ECOSYSTEM.md` Refinement 1: "headless (robots) is a CONFIG of
Jaeger AI, not a fork") + APP (whatever face the operator uses to talk
to the running instance — same faces as any other distribution
install).

## Framework gaps this example confirms (not new — cross-referenced)

Every gap named in `examples/two_axis_turret.md` (generalized
servo/joint contract) and the hardware-package/`module.yaml` gap named
in both the turret example and above is the SAME underlying gap JP01
itself hits — JP01 isn't exposing a new class of problem, it's the
real body proving the ones the hypothetical examples predict. See
`examples/FRAMEWORK_GAPS.md` for the deduped, tagged list.
